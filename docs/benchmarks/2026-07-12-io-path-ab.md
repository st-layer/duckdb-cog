# I/O 경로 실측: (a) object_store 직행 vs (b) DuckDB FileSystem 브릿지 — RFC §6.5

|  |  |
| -- | -- |
| **일자/환경** | 2026-07-12 · macOS arm64 (Apple Silicon) 단일 머신 |
| **대상** | cog **release 빌드** + `cog_io_bench()` 실험 함수 · duckdb-python 1.5.4 · httpfs(core) |
| **하네스** | `cog_io_bench(path, io := ...)` — **같은 엔진 워크로드**(open_cog 메타 → 산개 64점 read_pixels → 중앙 절반 창 zonal_stats)를 소스만 바꿔 실행. T5 counting 래퍼가 reader 경계의 fetch 횟수·바이트를 세고, 픽셀 체크섬으로 정합을 판정 |
| **소스** | `file`=seek/read · `object_store`=§6.5(a) 현행 · `duckdb_fs`=§6.5(b) 프로토타입 (DuckDB FileSystem C-API) |
| **방법** | cold(신규 커넥션·프로세스 캐시 없음) median of 5 (원격 3). 재현: `uv run python scripts/bench_io_compare.py local http remote secret` |

## 프로토타입이 확인한 실현 가능성

- **DuckDB 1.5.4 C-API 는 (b) 를 구현하기에 충분히 노출돼 있다**:
  `duckdb_table_function_get_client_context` → `duckdb_client_context_get_file_system`
  → `duckdb_file_system_open` / `duckdb_file_handle_{seek,read,size,close}`.
  httpfs 로드 시 http(s)·s3 경로가 그대로 열린다 (VirtualFileSystem 디스패치).
- **정합**: 5개 구성(로컬 3 + http 2) 전부에서 points/window 체크섬 동일,
  window 체크섬은 오라클(T1) 검증 경로인 `RS_ZonalStats` 'sum' 과 정확 일치 —
  `test/sql/io_bench.test` 가 E2E 계약으로 고정.
- **EOF 클램프 계약**(32KiB readahead 선요청 > 파일 크기)도 DuckDB FileHandle
  위에서 성립 — tiny 픽스처 E2E 통과.
- **요청 병합은 엔진 측 성질**이라 소스와 무관: 모든 구성에서 fetch 횟수·바이트가
  동일했다 (메타 1회/32KiB · 산개 64점 51회/6.7MB · 중앙 창 64회/8.4MB).
  → (b) 채택 여부는 병합 동작을 바꾸지 않는다. 차이는 순수하게 전송 계층 비용.

## 결과 — 로컬 파일

| io | wall | open | meta | points | window |
| -- | -- | -- | -- | -- | -- |
| file (현행 로컬) | 12.7 ms | 0.0 | 0.1 | 1.4 | 11.0 |
| object_store(`file://`) | 13.6 ms | 0.0 | 0.1 | 2.5 | 10.8 |
| **duckdb_fs (b)** | **10.7 ms** | 0.0 | 0.1 | 1.1 | 9.4 |

로컬은 사실상 동급 — (b) 가 근소 우위 (DuckDB LocalFileSystem 의 단순 pread).

## 결과 — 로컬 http (RangeHTTPServer, 결정적 대조)

| io | wall | open | meta | points | window |
| -- | -- | -- | -- | -- | -- |
| object_store (a) | 120.9 ms | **78.4** | 0.5 | **16.8** | 25.2 |
| **duckdb_fs (b)** | **47.5 ms** | **0.4** | 0.7 | **27.6** | 18.7 |

- (a) 의 open 78ms 는 object_store http 클라이언트 초기화 비용 (커넥션 풀·TLS
  스택 준비; 로컬 서버라 네트워크 지연이 아님) — 파일당 1회.
- 픽셀 단계는 혼조: points 는 (a) 우위, window 는 (b) 우위 — 요청 수가 동일하므로
  커넥션 재사용·버퍼링 전략 차이 수준. 총 wall 은 (b) 우위.

## 결과 — 실 원격 (Sentinel-2 B04, us-west-2)

| io | wall | open | meta | points | window |
| -- | -- | -- | -- | -- | -- |
| object_store (a) | 16.5 s | 0.17 s | **0.60 s** | **8.0 s** | 7.7 s |
| **duckdb_fs (b)** | **11.1 s** | 0.44 s | 0.80 s | 8.7 s | **0.95 s** |

(median of 3, cold. fetch 횟수·바이트는 두 경로 동일: 메타 1회/32KiB ·
points 46회/1.5MB · window 49회/101KB†)

- **콜드 메타는 (a) 근소 우위** (0.60 vs 0.80 s) — open 포함 총 첫-접촉도 (a) 0.77 vs (b) 1.24 s.
- **산개 포인트는 동급** (8.0 vs 8.7 s) — 46회의 이산 range 왕복이 지배, 전송 계층 차이는 오차 수준.
- **인접-창 읽기는 (b) 압승** (0.95 vs 7.7 s, 8×) — 같은 49회 요청인데 (a) 는 요청마다
  실 왕복(49 × ~150ms RTT ≈ 7.4s)인 반면, httpfs 는 내부 read 버퍼가 인접 range 를
  물리적으로 합쳐 왕복을 흡수한다. 우리 병합이 타일 단위로 끊는 인접 요청 패턴에서
  (b) 가 공짜 coalescing 을 얻는 구조 — #26(전역/연결 캐시) 이 (a) 에서 메꿔야 할 갭의 크기.

† 이 씬의 중앙 절반 창은 nodata 지배 영역이라 (8XMR 고위도 타일) 타일들이 사실상
빈 DEFLATE (~2KB/타일) — window 체크섬도 양쪽 다 0 (nodata 픽셀 합산 제외 규약,
패리티 자체는 성립). 유효 픽셀이 있는 points 체크섬(16265)은 양쪽 정확 일치.

## CREATE SECRET / AWS_SKIP_SIGNATURE (#21 footgun) 검증

(b) 경로, **환경변수 전무** 상태에서 공개 버킷(sentinel-cogs) 접근:

| 케이스 | 결과 |
| -- | -- |
| `https://` , secret 없음 | **OK** (meta 0.91 s) |
| `s3://` , secret 없음 | **OK** (meta 0.88 s) — httpfs 가 익명 unsigned 로 진행 |
| `s3://` + `CREATE SECRET (TYPE s3, PROVIDER config, REGION ...)` | **OK** (meta 0.79 s) |

**#21 의 footgun 은 (b) 에서 소멸한다.** (a) 는 virtual-host 스타일 URL 을 S3 로
인식하면 자격증명 부재 시 EC2 메타데이터 서비스를 폴링하며 hang → `AWS_SKIP_SIGNATURE=true`
를 사용자가 알아야 했다 (README 각주). (b) 는 아무 설정 없이 즉시 동작하고, 자격증명이
필요한 버킷은 DuckDB 표준 `CREATE SECRET` 생태계(credential_chain 포함)를 그대로 쓴다.

## (b) 의 엔지니어링 비용 (프로토타입에서 드러난 것)

1. **duckdb-rs 표면 공백**: `BindInfo` 가 raw `duckdb_bind_info` 를 감춘다 —
   프로토타입은 single-field struct `transmute_copy`(size 정적 단언 동반)로 우회.
   프로덕션화하려면 duckdb-rs upstream 에 접근자(또는 client context 노출) PR 이
   선행돼야 한다.
2. **스칼라 함수 경로 부재**: client context 는 table function bind 에서만 얻을 수
   있다 (duckdb-rs vscalar 표면에는 경로 없음). `RS_Value` 등 스칼라의 (b) 전환은
   별도 upstream 작업 없이는 불가 — 전환한다면 read_cog/read_stac(테이블 함수)만
   먼저 가는 이원화 기간이 생긴다.
3. **핸들 직렬화**: DuckDB FileHandle 은 seek+read stateful → Mutex 직렬화 필요.
   현 엔진은 요청을 순차 발행하므로 실측 손해는 없었지만, 향후 동시 fetch
   (tokio worker 증설, #26 전역 캐시의 프리페치)와는 구조적으로 상성이 나쁘다 —
   핸들 풀 또는 파일당 다중 핸들로 우회 가능하나 추가 복잡도.
4. **수명 제약**: context→filesystem→handle 을 bind 중에 열고 bind 안에서 다
   쓰는 현 프로토타입 구조는 안전하지만, 핸들을 쿼리 실행 단계(func)나 리더
   캐시로 넘기려면 수명·스레드 규약을 DuckDB 내부와 다시 맞춰야 한다.

## 권고 (결정 재료 — 채택 결정은 오너)

**측정이 말하는 것:**

1. **실현 가능성**: (b) 는 오늘 C-API 로 구현 가능하고 정합이 E2E 로 성립한다.
   "노출 불충분" 분기(#30 이 예정했던)는 발생하지 않았다 — 단, duckdb-rs 래퍼
   공백 2건(위 비용 1·2)은 실재한다.
2. **성능은 채택 반대 근거가 못 된다**: 로컬 동급, 원격은 혼조이나 총합에서 (b) 우위
   (인접-창 8×). (a) 가 확실히 이기는 건 콜드 첫-접촉 ~0.5s 뿐.
3. **UX 는 (b) 의 일방적 우위**: AWS_SKIP_SIGNATURE footgun 소멸 + CREATE SECRET
   단일화 + 프록시/캐시 등 DuckDB FS 생태계 편승.
4. **구조적 리스크는 동시성**: 핸들 직렬화가 향후 동시 prefetch(#26)와 상성이 나쁘다.
   핸들 풀로 우회 가능하나, 이건 (b) 전면 전환 시에만 부담이 되는 비용.

**옵션 정리 (채택 결정은 오너):**

- **O1. (a) 유지**: 현상 유지. footgun 은 README 각주로 계속 안고 간다. 비용 0.
- **O2. (b) 전면 전환**: UX 최선. duckdb-rs upstream PR(bind ptr/context 접근자,
  스칼라 경로) 선행 필요 — 스칼라까지 넘기기 전 이원화 기간 불가피.
- **O3. 하이브리드 (테이블 함수만 (b), 스칼라는 (a) 유지 / 또는 `io` 파라미터 opt-in)**:
  upstream 의존 없이 지금 가능(프로토타입 코드 재사용). 자격증명 이원화가 부분
  해소되는 대신 "함수마다 I/O 경로가 다르다"는 설명 부담이 생긴다.

엔지니어링 관점의 권고는 **O3 로 시작해 upstream 정리 후 O2 를 재평가** —
근거: UX 이득(3)은 원격 사용자 진입 장벽 자체이고, O3 의 비용은 문서 1절이며,
O2 의 유일한 실질 블로커(1)는 우리 통제 밖(upstream 머지 일정)이다.
