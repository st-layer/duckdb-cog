# 시계열 추출 벤치마크: cog vs PostGIS vs raquet vs duckdb-raster

|  |  |
| -- | -- |
| **일자/환경** | 2026-07-23 · macOS arm64 (Apple Silicon) 단일 머신, cog **release 빌드** |
| **질문** | "모든 후보가 이미 셋업된 상태"라면, 영역 시계열을 뽑는 순수 속도는 누가 빠른가 |
| **데이터** | 합성 12씬 × 2048² u16 DEFLATE COG (256px 타일, EPSG:32652, seed 42 — 씬별 위상 변화로 시계열 신호 포함) |
| **워크로드** | 고정 창(1024² px = 10.24×10.24 km) zonal **mean** × 12씬 → 시계열 |
| **방법** | 전 후보 warm(사전 1회 실행 후), median of 5 (RT 는 3). 재현: `scripts/bench_timeseries.py` |
| **셋업 전제** | PostGIS 는 임포트+GiST 인덱스 완료, raquet 은 재인코딩 완료 상태에서 측정 — 셋업 비용은 하단 별표 |

## 결과 — 12씬 시계열 추출

| 후보 | 시간 | 상대 | 값 정합 |
| -- | -- | -- | -- |
| **cog** (`RS_ZonalStats` SQL 한 방) | **95.4 ms** | 1× | 정확 (창 = 정확히 1024², 픽셀 중심 규약) |
| PostGIS (`ST_SummaryStatsAgg(ST_Clip(...))`, 서버측 시간) | 272.9 ms | 2.9× | 근접 — mean 0.2% 차이는 경계 규약† |
| raquet (quadbin 블록 정렬 창, gzip decode) | 363.8 ms | 3.8× | **재인코딩으로 픽셀값 변형**‡ |
| duckdb-raster (`RT_CubeClip`+`RT_CubeStats_Agg`) | 3,322.7 ms | 35× | **정합 실패 — fill 값 반환**§ (참고치) |

† **경계 규약 차이 실측**: 같은 창에서 PostGIS `ST_Clip` 은 경계에 '교차'하는
픽셀까지 포함해 **1,052,676 = 1026²** 픽셀을 집계하고, 우리는 픽셀 중심 포함
규약으로 **정확히 1,048,576 = 1024²** 를 집계한다. 둘 다 정당한 규약이나 창
정의에는 우리 쪽이 엄밀하며, mean 차이(3499.4 vs 3492.6, 0.2%)는 전부 이
한 겹의 경계 픽셀에서 온다.

‡ raquet 은 UTM → WebMercator quadbin **재투영·리샘플링·재인코딩**을 거친
데이터를 읽는다 — 시계열 값(3496~3517)이 원본 픽셀의 정확한 함수가 아니다
(N2 논점). 창도 quadbin 블록 정렬 근사(zoom 14, 36블록). 참고: 2026-07-12
비교벤치에서 "변환 도구 미가용"이었던 raquet-io 는 pip `gdal` 을 시스템
libgdal 버전(3.11)에 핀하면 동작함을 확인 — 이번에 처음 실측 가능해졌다.

§ `RT_CubeClip` 은 band 인자 0-based 로 호출해야 실행되며(1-based 는 에러),
실행돼도 mean=0.0 · valid_count=2359296(=2304², 창 클리핑도 미적용)으로
**fill 값을 집계**한다 — 2026-07-12 비교벤치의 정합 실패가 그대로 재현.
수치는 "그 모델의 연산 비용" 참고치로만 싣는다.

## 셋업 비용 (같아지기까지의 값)

| | cog | PostGIS | raquet | duckdb-raster |
| -- | -- | -- | -- | -- |
| 사전 준비 | **0** (파일 그대로) | 임포트 37 s + docker 상주 | 재인코딩 17 s (+값 변형) | 0 |
| 부대 조건 | — | 서버 프로세스·클라이언트 도구 | GDAL 파이썬 바인딩 (버전 핀 필요) | — |

## 해석

1. **상대에게 최대한 유리한 전제(전부 셋업 완료·웜)에서도 cog 가 2.9~3.8× 빠르다.**
   창이 걸치는 타일(씬당 16개)만 fetch+decode 하는 부분 순회 설계가, 서버
   왕복·전체 클립(PostGIS)이나 블록 전체 gzip decode(raquet)를 이긴다.
2. **셋업 비용까지 더하면 격차는 구조적**: "파일 도착 → 첫 시계열"의 총비용에서
   경쟁 후보들은 수십 초의 선행 작업 + 상주 인프라(docker)나 값 변형(raquet)을
   지불한다. cog 는 0.
3. **정확성 축은 이번에도 갈렸다**: cog 는 창 정의가 픽셀 단위로 정확하고
   (rasterio 오라클로 CI 검증), PostGIS 는 근접(규약 차이), raquet 은 값 자체가
   변형, RT 는 실패. 속도와 정확성이 같은 편에 선 후보는 cog 뿐이다.
